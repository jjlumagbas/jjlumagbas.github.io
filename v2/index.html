Case study

Quality People: User flow analysis and design for a new dashboard feature

Quality People uses clever marketing and big-data analytics to help schools find teachers who best fit open positions. It allows schools to launch targeted marketing to job-seekers, and then usher applicants through their hiring pipeline.

I helped design and launch a feature to allow schools to manage separate hiring pipelines for different job openings (then, the platform allowed for only a single pipeline). This required changes to multiple frontends, as well as the underlying data model and backend API.

Client Quality People, a platform that helps schools recruit teachers using big-data analytics and targeted marketing

Challenge Major feature update that required changes to multiple front-ends, as well as the underlying data model and backend API

My role Product Management: Document existing user flows and design updated user flows, UI sketches, and mockups; Lead implementation, manage remote developers, assemble and maintain sprint backlogs

Extra challenge Distributed team: The company is based in Austin, I worked remotely from South Texas, and I managed devs who operated out of the Philippines.

Extra extra challenge The original devs had moved on from the company! So initial work was needed to document existing user flows, to effectively onboard new set of developers

Outcome We successfully implemented and deployed the feature update, which opened up the platform to a significantly broader set of customers (schools)

Documenting existing user flows

But first...

Before we could start designing the new feature, we first needed a clear picture of existing user flows.

Since the original devs had already moved on from the company, and we were onboarding new devs, plus the team was distributed across two timezones, written (or in this case, drawn) documentation that could serve as reference for the team was critical.

To get a solid picture of the existing user flows, I:

  • conducted multiple interviews with the VP Product (the remaining member of the original team),
  • sent questions to the original devs (who at that point were working at new jobs),
  • did some code-diving (my CS background proved useful here), and
  • stepped through the platform as a user to see which buttons led where.

UI Changes

(Not) reinventing the wheel

With a good understanding of the existing (and target) user flows, design of the actual screens to support the change turned out to be fairly straightforward. I wrote user stories, and drew up sketches and mockups to serve as specs for the devs.

Problem What's the simplest way to allow a school hiring manager to view a different hiring pipeline, given he already had a Trello-like dashboard to show a single hiring pipeline?

Solution A project-switcher dropdown.

Lots of other software platforms do this: for instance, Trello for multiple boards (left), and Gmail for multiple accounts (right). It's a common enough pattern, and seemed intuitive enough for users to adopt.

As a bonus, the pattern is flexible enough to keep the door open for future feature updates, for instance:

  • To restrict view rights for various users, just filter the list of pipelines that are visible for that user.
  • Will the user want to view multiple pipelines at once? Add checkboxes to each pipeline in the dropdown, and allow users to select multiple pipelines.

Reflection sidebar

A few paragraphs back I said this UI change seemed intuitive enough for users to adopt. How did I know this? Two possible answers:

I Don't! I'm not the user, so this is definitely an assumption I'm making. Ideally, we would have done usability tests on mockups with a few actual users to validate assumptions before doing any actual implementation. In this case, the team bias was more towards shipping a likely good enough solution, and so we skipped usability testing.

Heuristics This decision, however, was not at all merely a shot in the dark: Jakob Nielsen says users spend most of the time on other sites, and so using a common UI pattern had a good chance of succeeding at communicating behavior.

Outcomes

Vision to actionable specs

I successfully bridged the gap between VP Product, who carried the product vision, and translate that vision into feature specs for the development team.

We did, however, experience some development delays. Note for next time: ensure devs are actively involved in creating sprint targets, and have them own the backlog!

In the end, we were able to implement and deploy the multiple-pipeline feature, opening up the platform to a significantly broader set of school customers.

Lessons learned

A diagram = 1K words

For me, this project highlighted the ability of visual documentation to be an efficient repository of information: they're quick to produce, you can pack in a lot of information, and with enough shared understanding they are evocative for the reader without being verbose.

For any details that need clarifying, you can just hash it out in your project management tool (Github Projects, in our case).
